# Global Ocean Dynamic Adjustment

date: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

## Summary

This design document describes a new Polaris capability for dynamic adjustment
of global-ocean initial conditions. The intent is to port the core idea from
Compass' `compass/ocean/tests/global_ocean/dynamic_adjustment` workflow: a
sequence of short MPAS-Ocean forward runs with stage-specific time steps,
damping, and restart/output intervals that dissipate fast waves generated by
the initial condition before a longer production simulation begins.

The new Polaris capability should support the same qualitative workflow for
both MPAS-Ocean and Omega. A developer should be able to describe a staged
adjustment schedule over several days to roughly a month, with progressively
less damping and typically increasing time steps, and Polaris should turn that
schedule into an inspectable sequence of model steps. Each stage should produce
restart files and optional diagnostics so the developer can verify that the solution
is settling down and can continue from the final relaxed state.

The primary software challenge is not the numerical method itself but the
workflow abstraction: Polaris needs a reusable way to describe and execute a
chain of forward model stages while mapping common stage settings onto
model-specific MPAS-Ocean and Omega configuration options. This design is
successful if Polaris can express the existing Compass-style schedules in a
compact configuration file, execute them as a restart chain, and leave a clean
path for model-specific tuning and validation as the capability matures.

## Requirements

### Requirement: A staged dynamic-adjustment workflow can be created from a global initial condition

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Polaris shall provide a workflow that begins from a global-ocean initial
condition and runs a sequence of forward-model stages intended to dissipate
fast external and internal waves introduced by initialization.

The workflow shall support schedules spanning from a few days to roughly a
month, depending on mesh resolution and model stability needs.

Each stage shall support its own run duration and restart/output cadence and
shall allow stage-specific control over time stepping and damping.

### Requirement: The same conceptual workflow supports both MPAS-Ocean and Omega

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Polaris shall support dynamic adjustment for both MPAS-Ocean and Omega through
a common workflow description rather than two unrelated task designs.

The common description shall allow Polaris to represent concepts such as stage
duration, time-step selection, damping strength, and restart chaining even if
the exact model configuration options differ between MPAS-Ocean and Omega.

### Requirement: Dynamic adjustment is decomposed into inspectable restart stages

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

The workflow shall be organized into individual Polaris steps or substeps, one
per stage of the dynamic-adjustment schedule.

Each stage shall consume either the original initial condition or the restart
from the preceding stage and shall produce a restart suitable for the next
stage. The final stage shall produce a relaxed restart that can be used for
subsequent simulation workflows.

### Requirement: Adjustment schedules are reusable but easy to tune

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Polaris shall support reusable default schedules for common global meshes while
also allowing users or developers to modify durations, damping values, and
time steps without changing Python source code.

The design shall make it straightforward to port existing Compass schedules
into Polaris and later refine them as model behavior or stability limits
change.

### Requirement: The workflow includes basic validation that the adjustment remained well behaved

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Polaris shall provide at least basic validation that the dynamic-adjustment
sequence completed successfully and did not produce obviously unphysical
results.

This validation does not need to prove that the schedule is scientifically
optimal, but it should detect clear failures such as numerical blow-up, missing
restart chaining, or gross excursions in key diagnostics.

## Algorithm Design

### Algorithm Design: A staged dynamic-adjustment workflow can be created from a global initial condition

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

The workflow should be described by an ordered schedule, likely in YAML,
containing a small number of shared options plus a list of stages. Each stage
should contain:

1. A stage name.
2. A run duration.
3. An output interval.
4. A restart interval.
5. Time-step information, either explicit or derived from mesh resolution.
6. Damping settings, typically decreasing from one stage to the next.

This is directly analogous to the existing Compass design, where meshes provide
YAML files containing stage sequences such as several `damped_adjustment_*`
stages followed by a final `simulation` stage. Polaris should preserve this
basic mental model because it is already easy to understand and has proven
useful across several global meshes.

### Algorithm Design: The same conceptual workflow supports both MPAS-Ocean and Omega

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

The schedule should be model-agnostic. Polaris should interpret the schedule
into a common internal representation and then map each stage into model
configuration options appropriate for the selected ocean model.

For MPAS-Ocean, this mapping will likely include familiar options such as
`config_dt`, `config_btr_dt`, restart flags, and the Rayleigh damping controls
used in Compass. For Omega, Polaris should provide a model-specific translator
that applies the closest available controls for time stepping, restart
behavior, output cadence, and damping.

The initial implementation does not need to guarantee that MPAS-Ocean and Omega
use identical low-level damping controls. The requirement is instead that both
models can participate in the same staged-adjustment workflow abstraction and
that model-specific differences are isolated in one place.

### Algorithm Design: Dynamic adjustment is decomposed into inspectable restart stages

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Polaris should create one model-running step per adjustment stage. The first
stage should start from the initial condition produced by the corresponding
global-ocean initialization workflow. Each later stage should start from the
restart written by the previous stage.

This stepwise structure makes the workflow easy to inspect and debug. If a
particular stage is unstable or clearly overdamped, the user can examine that
stage's configuration and outputs without having to reason about a single large
monolithic run script.

The workflow should also compute each stage's start and stop time from the
ordered schedule so restart filenames and model start times are deterministic
and do not need to be entered redundantly by hand.

### Algorithm Design: Adjustment schedules are reusable but easy to tune

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

The most practical near-term design is to store default schedules in small
YAML files associated with a mesh, task, or mesh family, following the same
pattern Compass already uses.

Polaris should support two complementary modes:

1. A default mesh-specific schedule that can be checked into the repository.
2. User overrides through Polaris config options or an alternate schedule file.

This keeps the common case simple while avoiding hard-coded stage definitions
in Python.

### Algorithm Design: The workflow includes basic validation that the adjustment remained well behaved

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Validation should focus on clear failure modes rather than exhaustive physical
assessment. A likely initial strategy is:

1. Confirm that each stage completed and wrote the expected restart.
2. Confirm that the final stage produced the expected output file.
3. Record a small set of end-of-stage summary diagnostics so the last several
   stages can be inspected automatically rather than only by manual plotting.
4. Check simple global diagnostics or summary statistics against loose
   thresholds to detect numerical blow-up.
5. For at least the last three stages, verify that selected adjustment metrics
   are flattening or decreasing, for example maximum cell kinetic energy,
   domain-integrated kinetic energy, or sea-surface-height tendency.
6. Optionally compare a small set of final-state variables against a baseline
   for regression testing.

This is similar in spirit to the existing Compass validation, which checks
selected output variables and simple thresholds from global statistics.

## Implementation

### Implementation: A staged dynamic-adjustment workflow can be created from a global initial condition

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Polaris should add a new global-ocean dynamic-adjustment task that consumes the
outputs of the global-ocean initialization workflow. A likely structure is a
task or test case that reads a schedule file and instantiates one forward model
step per stage.

The schedule format can initially stay very close to the Compass format to
reduce porting effort. For example:

```yaml
dynamic_adjustment:
  get_dt_from_min_res: false
  steps:
    damped_adjustment_1:
      run_duration: 02_00:00:00
      output_interval: 10_00:00:00
      restart_interval: 02_00:00:00
      dt: 00:05:00
      btr_dt: 00:00:15
      Rayleigh_damping_coeff: 1.0e-4
    simulation:
      run_duration: 10_00:00:00
      output_interval: 10_00:00:00
      restart_interval: 10_00:00:00
      dt: 00:10:00
      btr_dt: 00:00:15
      Rayleigh_damping_coeff: None
```

Internally, Polaris may later rename `steps` to `stages` or introduce a more
structured schema, but preserving the Compass layout initially will make the
first port much easier.

### Implementation: The same conceptual workflow supports both MPAS-Ocean and Omega

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Implementation should build on Polaris' existing model-abstraction machinery,
especially `OceanModelStep` and the existing mapping between common YAML-driven
configuration and model-specific MPAS-Ocean or Omega settings.

The dynamic-adjustment code should therefore separate:

1. Schedule parsing and restart sequencing.
2. Common stage metadata such as duration and cadence.
3. Model-specific translation of time-step and damping options.

This separation should keep the workflow portable even if Omega ends up needing
different config names or a different subset of supported damping controls than
MPAS-Ocean.

### Implementation: Dynamic adjustment is decomposed into inspectable restart stages

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Each stage should live in its own work directory and should declare:

1. The restart or initial-condition input it consumes.
2. The restart file it produces for the next stage.
3. A stage-local output file for diagnostics or later validation.

The Python driver should compute restart times from cumulative run durations so
the user does not have to provide both a stage duration and an absolute stop
time. This is another area where the existing Compass helper logic can be
ported almost directly.

### Implementation: Adjustment schedules are reusable but easy to tune

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Default schedules should likely live with the relevant global mesh or task
package. This makes them version-controlled, easy to review, and easy to tune
for a specific resolution family.

For the initial implementation, it is sufficient to support a small number of
checked-in schedules rather than inventing a sophisticated automatic tuning
procedure. The important design point is that the schedule is data-driven and
not embedded in Python logic.

### Implementation: The workflow includes basic validation that the adjustment remained well behaved

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Validation should include lightweight checks in the task or test case, such as
verifying expected restart files, confirming the final output exists, and
checking a small set of variables or summary statistics against baseline or
threshold values.

The workflow should also produce a compact machine-readable summary of
stage-end diagnostics, for example one row per stage with metrics such as
maximum cell kinetic energy, domain-integrated kinetic energy, and one or more
simple sea-surface-height or tracer tendencies if available from both models.
This summary would support both automated checks and quick visual inspection at
the end of the run.

An initial heuristic check could require that the maximum cell kinetic energy
at the end of each of the last three stages decreases monotonically, or at
least does not increase by more than a small tolerance. Similar checks could
be applied to one or two other globally aggregated measures once experience
shows which diagnostics are the most robust indicators of a settling state.

Where MPAS-Ocean and Omega diagnostics differ, the validation API should still
present a common concept of "stage completed successfully and final adjusted
state looks reasonable," with model-specific implementations underneath.

## Testing

### Testing and Validation: A staged dynamic-adjustment workflow can be created from a global initial condition

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Unit tests should cover schedule parsing, cumulative restart-time calculation,
and error handling for malformed schedules.

An integration test should confirm that a short multi-stage adjustment can be
set up and run on a coarse global mesh.

### Testing and Validation: The same conceptual workflow supports both MPAS-Ocean and Omega

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Model-specific smoke tests should confirm that the same staged schedule can be
translated into runnable MPAS-Ocean and Omega workflows, even if the resulting
low-level model options are not identical.

### Testing and Validation: Dynamic adjustment is decomposed into inspectable restart stages

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Regression tests should verify that each stage writes the restart needed by the
next stage and that the final stage produces the expected relaxed restart and
diagnostic output files.

### Testing and Validation: Adjustment schedules are reusable but easy to tune

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

At least one checked-in schedule should be exercised in regression testing so
future changes to parser logic or model-option mapping do not silently break
the default workflow.

### Testing and Validation: The workflow includes basic validation that the adjustment remained well behaved

Date last modified: 2026/03/22

Contributors: Xylar Asay-Davis, Codex

Regression testing should include loose threshold checks or baseline
comparisons on a few key fields or diagnostics to catch obvious failures such
as numerical instability, missing restart chaining, or grossly unphysical
state variables.

Where practical, these tests should also confirm that the end-of-stage summary
for the last few stages shows tapering behavior in at least one kinetic-energy
metric, so that a schedule that is technically stable but clearly not settling
down is easier to catch.
